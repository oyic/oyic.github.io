# SEO Blog on chrisalberto.site, Populated by auto-linkedin

Date: 2026-07-10
Status: Approved (design), not yet implemented
Repos involved:
- `oyic.github.io` (this repo, static site, GitHub Pages, custom domain `chrisalberto.site`)
- `auto-linkedin` (`/Users/calberto/code/auto-linkedin`, Next.js on Vercel, Postgres via Drizzle, Vercel Workflow)

## 1. Goal

Every LinkedIn post generated by auto-linkedin already comes with a full long-form
article (`posts.articleTitle` / `posts.articleText`) — the LinkedIn post text is
literally derived from it. Today that article is thrown away after the LinkedIn post
ships. This project publishes that article as a real, SEO-indexable page under
`chrisalberto.site/blog/`, and links back to it from the LinkedIn post itself, so each
LinkedIn post drives traffic/backlinks to the portfolio site.

Non-goals:
- No new content generation. The article already exists; we're just publishing it.
- No CMS, no server-rendered blog, no database-backed portfolio site. The portfolio
  stays a zero-build static site.
- No human review/approval gate before a post goes to the blog (confirmed: fully
  automatic, same trust level as the existing LinkedIn auto-publish).

## 2. Architecture / data flow

```
auto-linkedin (Vercel)                          oyic.github.io (GitHub Pages, chrisalberto.site)
─────────────────────                          ──────────────────────────────────────────────
posts table already has:
  articleTitle, articleText  (full article, AI-generated)
  articleUrl                 (currently unused column — becomes the blog link)

publish-post workflow (runs at scheduledAt):
  1. getAuthorizedClient()
  2. fetchPostContent()
  3. NEW: publishArticleToBlog(post)  ─────────▶  one atomic commit containing:
        - render HTML from portfolio's              /blog/<slug>/index.html         (new post page)
          blog template (fetched from repo           /blog/index.html                 (listing, new entry
          at publish time)                                                             prepended between markers)
        - download the post's visual image           /sitemap.xml                      (new <url> entry,
          from Vercel Blob                                                              marker-based insert)
        - commit via GitHub Contents/Git             /feed.xml                          (new <item>, marker-
          Data API, get the resulting URL                                               based insert)
        - save articleUrl on the post                /assets/blog/<slug>/cover.<ext>   (downloaded image)
  4. append "Read the full article: <url>" to commentary
  5. uploadImageToLinkedin()   (unchanged — still posts the native image to LinkedIn too)
  6. publishToLinkedin(commentary with link)
  7. markPublished()
```

The blog push happens **before** the LinkedIn publish call, because the LinkedIn
commentary needs the live URL. If the blog push fails, the step throws, the existing
`try/catch` in `publishPostWorkflow` catches it, and the post is marked `failed` (same
path as an auth failure today) — no LinkedIn post goes out without its blog page
existing, since the backlink is the point of this feature.

## 3. auto-linkedin changes

### 3.1 New module: `lib/blog/publish-to-blog.ts`

Exposes one function, called as a workflow step (`"use step"`) from
`workflows/publish-post.ts`:

```ts
async function publishArticleToBlog(post: Post, visual: Visual | undefined): Promise<{ articleUrl: string }>
```

Responsibilities:

- **Slug generation**: kebab-case `post.editedArticleTitle ?? post.articleTitle ?? post.topic`,
  truncate to ~60 chars at a word boundary, append a short suffix derived from
  `post.id` (e.g. first 6 hex chars) to guarantee uniqueness without needing to query
  the other repo for collisions. Pure function, unit-testable in isolation (same
  pattern as `derive-linkedin.ts`).
- **Template fetch**: GET `blog/_template.html` and the listing/sitemap/feed marker
  files from `oyic.github.io` via the GitHub Contents API at publish time, so the
  portfolio repo remains the single source of truth for blog page design — a CSS/markup
  tweak there doesn't require redeploying auto-linkedin.
- **Image**: fetch bytes from `visual.blobUrl` (if a visual exists for the post),
  base64-encode for the GitHub commit. If no visual exists, the post ships without a
  featured image (falls back to the site's default OG image).
- **Commit**: build one atomic commit via the GitHub Git Data API (create blobs →
  create tree → create commit → update `refs/heads/main`) touching:
  - `blog/<slug>/index.html` (new file)
  - `blog/index.html` (existing file, new `<article>` teaser inserted between
    `<!-- BLOG_LIST_START -->` / `<!-- BLOG_LIST_END -->` markers)
  - `sitemap.xml` (existing file, new `<url>` block inserted between
    `<!-- SITEMAP_URLS_START -->` / `<!-- SITEMAP_URLS_END -->` markers)
  - `feed.xml` (existing file, new `<item>` inserted between
    `<!-- FEED_ITEMS_START -->` / `<!-- FEED_ITEMS_END -->` markers, list capped to the
    most recent 20 items)
  - `assets/blog/<slug>/cover.<ext>` (new file, only if a visual exists)

  One commit per published post keeps the portfolio repo's history clean and each
  publish atomic (either all four files land, or none do).
- **Auth**: a fine-grained GitHub PAT scoped only to `oyic.github.io` with
  `contents: write`, stored as `GITHUB_PORTFOLIO_TOKEN` in Vercel's env vars for the
  auto-linkedin project. Never logged; only used inside this module.

### 3.2 `db/queries.ts`

`updatePost`'s patch type already restricts to a known set of columns; add
`articleUrl` to that allowed set (the column exists on `posts` already, just unused).

### 3.3 `workflows/publish-post.ts`

Insert the new step between `fetchPostContent` and `uploadImageToLinkedin`:

```ts
const { articleUrl } = await publishArticleToBlog(post, visual);
await updatePost(postId, { articleUrl });
const commentaryWithLink = `${commentary}\n\nRead the full article: ${articleUrl}`;
```

`commentaryWithLink` replaces `commentary` in the subsequent `publishToLinkedin` call.
If `publishArticleToBlog` throws, it propagates to the existing outer `try/catch`,
which calls `markFailed` — consistent with how auth/publish failures are already
handled.

### 3.4 Dashboard (`app/posts/[id]/post-editor.tsx`)

Once `postStatus === "published"` and `post.articleUrl` is set, render a "View on
blog →" link next to the status badge, pointing at `post.articleUrl`.

## 4. portfolio (`oyic.github.io`) changes

New structure:

```
blog/
  _template.html          ← single-post template with placeholders:
                             {{title}}, {{description}}, {{body}}, {{image}}, {{imageAlt}},
                             {{date}}, {{isoDate}}, {{slug}}, {{canonicalUrl}}
  index.html               ← listing page, newest first, marker comments for inserts
  <slug>/index.html         ← one per published post (clean URL: /blog/<slug>/)
assets/blog/<slug>/cover.<ext>
sitemap.xml                 ← existing file, gains a marker block for blog URLs
feed.xml                    ← new file, Atom/RSS feed, marker block for items
```

- **Design consistency**: `_template.html` links the existing `assets/css/style.css`
  and self-hosted fonts (no duplicated CSS) — just header/nav/footer chrome plus an
  `<article>` block, matching the current warm-dark terminal aesthetic. A "Blog" link
  gets added to the nav in `index.html` (the homepage).
- **Per-post SEO**: each generated page gets its own `<title>`, meta description
  (derived from the article's opening line), `<link rel="canonical">`, OG/Twitter tags
  (using the downloaded cover image, or the site's default OG image if none), and
  `BlogPosting` JSON-LD (`headline`, `datePublished`, `image`, `author`) — mirroring
  the pattern already present in the homepage `<head>`.
- **Clean URLs**: GitHub Pages serves `blog/<slug>/index.html` at `/blog/<slug>/`
  without any server config, consistent with the site's existing no-build-step
  approach (no Jekyll, no `.nojekyll` concerns since there's no underscore-prefixed
  directory in this scheme).
- **`sitemap.xml`**: existing single-URL file gains a marker block; each publish
  inserts one more `<url>` entry with `lastmod` set to the publish date.
- **`feed.xml`**: new file, standard RSS 2.0 (or Atom) feed capped at the 20 most
  recent posts, linked from `<head>` via `<link rel="alternate" type="application/rss+xml">`
  on both the homepage and blog listing page.
- **`robots.txt`**: no change needed — already `Allow: /` for all crawlers including
  the AI-crawler allowlist, which will pick up `/blog/` automatically.

## 5. Error handling

- GitHub API failures (rate limit, 409 conflict on the ref update, network error)
  inside `publishArticleToBlog` throw a plain `Error`, caught by the existing
  `try/catch` in `publishPostWorkflow`, resulting in `status: "failed"` with
  `failureReason` set — visible in the dashboard exactly like existing failure modes.
- A 409 conflict on `refs/heads/main` (e.g. two posts publishing in the same instant)
  is treated as a transient failure — no built-in retry loop; a failed post can be
  manually re-scheduled from the dashboard, matching current behavior for any other
  publish failure.

## 6. Security

- `GITHUB_PORTFOLIO_TOKEN` is a new secret: a fine-grained GitHub PAT scoped only to
  the `oyic.github.io` repository with `contents: write` permission (no other repo
  access, no admin/workflow scopes). Stored as a Vercel environment variable on the
  auto-linkedin project, same handling tier as the existing LinkedIn OAuth secrets.

## 7. Testing

- Unit tests for slug generation and template placeholder filling (pure functions),
  following the existing `derive-linkedin.test.ts`-style pattern.
- A thin integration test for the GitHub commit call with a mocked GitHub API,
  following the existing `*.integration.test.ts` pattern already used for
  `publish-post.ts`.

## 8. One-time manual setup (not part of the automated pipeline)

- Create the fine-grained GitHub PAT and add `GITHUB_PORTFOLIO_TOKEN` to Vercel.
- Create the initial skeleton files in `oyic.github.io` with empty marker blocks:
  `blog/index.html`, `blog/_template.html`, `feed.xml`, and the marker comments added
  to `sitemap.xml`.
- Add the "Blog" nav link to the homepage.
